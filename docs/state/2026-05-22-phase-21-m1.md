## What shipped (private repo)

### Runtime patches (3 files)

1. **`src/bundle/defaultBundle.ts:shippedBundlePath()`** — extended with a `process.execPath`-relative discovery tried before the existing source-mode `import.meta.url` walk. The function takes optional `{ execPath, metaUrl }` test seams; production passes nothing. Source-mode behavior preserved bit-for-bit because the binary-mode check fails by design (no `bundle-default/` sibling next to bun).

2. **`src/cli/tuiLauncher.ts:findTuiBinary()`** — same pattern: `SOV_TUI_BIN` env var first (test seam + power-user escape), then binary-mode `process.execPath`-sibling check, then source-mode walk. Mirrors `shippedBundlePath`'s layered fallthrough.

3. **`src/version.ts`** — switched from `readFileSync(PKG_PATH)` to `import pkg from '../package.json' with { type: 'json' }`. The runtime-read approach broke under Bun `--compile` because `import.meta.url` resolves to `/$bunfs/` and `package.json` isn't part of the embedded FS. The JSON import freezes the version into the bundle at compile time. The `.bun-tag` + git-rev-parse SHA-suffix resolvers continue to work in source mode and gracefully return null in binary mode.

4. **`src/cli/upgrade.ts`** — added `detectInstallMode({execPath, homedir})` predicate, `BINARY_INSTALLER_URL` constant, `InstallMode` type, `mode?` field on `UpgradeOpts`. `buildUpgradeCommands` + `shouldPurgeCache` + `runUpgrade` all consult mode (auto-detect via process.execPath if unset). Binary mode short-circuits cache management (Bun's cache is irrelevant when we're not invoking Bun).

## Behavioral notes worth knowing next session

1. **Binary install lives at `~/.sov/`.** Layout: `bin/sov`, `bin/sov-tui`, `bundle-default/`, `version`. PATH-appended via `~/.zshrc` or `~/.bashrc` by `install.sh` (idempotent, grep-guarded).
2. **`sov upgrade` auto-detects mode** via `process.execPath.startsWith(${homedir}/.sov/bin/)`. Override with `opts.mode` (programmatic only — no CLI flag added).
3. **Bun `--compile` works with `bun:sqlite`** (validated via pre-flight spike). No DB-path special-casing needed.
4. **Source mode is unchanged.** Every existing source-mode invocation (`bun src/main.ts`, `bun install -g git+ssh://...`) hits the binary-mode checks, finds nothing relevant, falls through to the existing behavior bit-for-bit. Existing tests + workflows continue to pass.
5. **Release script needs `SOV_RELEASES_PATH`** pointing at a local clone of `yevgetman/sov-releases` (where `LICENSE.txt` is sourced from). The script dies cleanly with a helpful message if unset.
6. **macOS Gatekeeper quarantine** — surprisingly, the binaries downloaded via `curl | bash` on the dev host did NOT receive the `com.apple.quarantine` attribute (likely because curl in a script-pipe context doesn't trigger LaunchServices' quarantine policy). The installer still prints the `xattr -d` workaround as a precaution. First-time `curl | bash` users may or may not hit Gatekeeper depending on macOS version + Gatekeeper config; Apple Dev signing is the Phase 21 follow-up.
7. **`process.execPath` is the canonical handle** for binary-install resource discovery. Any new code that needs to find sibling assets should use this pattern: try `dirname(realpathSync(process.execPath))`-relative first, fall through to source-mode logic.
8. **JSON imports work in `--compile` mode.** Any future "load this static JSON / YAML config from src/ tree" need should prefer `import x from './foo.json' with { type: 'json' }` over runtime `readFileSync` for the same reason version.ts now does.
