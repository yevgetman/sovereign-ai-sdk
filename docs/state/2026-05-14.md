## Behavioral notes worth knowing next session

1. **Compaction creates a new session id; the TUI hops `m.sessionID`.** Per ADR M6-01: every compaction (proactive, overflow recovery, or explicit `/compact`) creates a fresh child session id. The TUI updates `m.sessionID` on receiving either the `compaction_complete` SSE event OR the JSON response from `POST /sessions/:id/compact`. Subsequent POSTs (turns, approvals, further compactions) route to the new child. SessionDb persists parent→child lineage via `recordCompactionLineage` for audit; the SSE subscription stays on the parent id (the bus is keyed on parent and continues to surface post-compaction events under the parent id).

2. **Single retry on context-overflow.** Per ADR M6-02: first overflow → compact + retry once. Second overflow → `turn_error`. The proactive + recovery branches operate on independent budgets — a proactive compaction earlier in the same turn does NOT prevent the recovery branch from firing, so a single turn can emit TWO `compaction_complete` events. The TUI's compaction handler is idempotent — it pivots `m.sessionID` to the LATEST `activeSessionId` (the recovery child id wins over the proactive child id when both fire).

3. **`POST /sessions/:id/compact` is synchronous.** Per ADR M6-03: the explicit verb runs compaction inline and returns the JSON `CompactResult` body. No SSE event fires for this path — the JSON response IS the notification. Errors return JSON-shaped responses (400/404/500) with `{ error: <message> }` envelope matching sibling routes.

4. **Microcompaction triggers based on tool-result token share.** `shouldMicrocompact(history, config)` returns true when the per-part `tool_result` blocks exceed `config.triggerThresholdPct` of the estimated total context. Default 40%. The microcompact pass replaces older tool_result content with `[Tool result cleared`-prefixed placeholders, keeping the most recent `config.keepRecent` (default 5) intact. Fires inside the turn loop AFTER `runTools` and BEFORE the next provider call — mutates the in-flight history array but never persists. Default `compactableTools` covers Bash + WebFetch + Read; configurable via `userSettings.microcompaction` (though `compactableTools` is not in the schema today — M6 left it as a code default).

5. **Proactive threshold stored as percentage.** `userSettings.compaction.proactiveThresholdPct` is a 1..99 integer (human-friendly editing); `buildRuntime` divides by 100 to get the fraction `shouldCompactProactively` expects. Default 0.75 from the function's own default. The compactor self-guards against the frozen system prompt alone exceeding the threshold (`compactor.ts:177-183`) — without that guard, compaction would loop indefinitely because the child still has the same system prompt.

6. **Same-provider summarization (no auxiliary model).** Per inline decision M6-06: the `summarize` callback uses `runtime.resolvedProvider.transport.stream(...)` with `runtime.model`. No auxiliary cheap-model routing. Auxiliary selection is Phase 15 / M8 polish.

7. **The wrapped-runtime test pattern.** The M6 integration suite (`tests/cli/tuiLauncherIntegration.test.ts:M6 describe block`) uses a `buildWrappedRuntimeModule` factory to inject `pre`/`post` hooks before/after `buildRuntime` runs. The wrapper captures the function reference BEFORE the mock is mounted so the wrapper's recursion into "the real" `buildRuntime` resolves to the production implementation rather than re-entering itself (avoids stack overflow). This pattern is the right shape for any future integration test that needs to inject behavior the launcher's CLI-shaped opts bag doesn't expose.

8. **The microcompact event is NOT yet on the wire.** `query()` emits a `microcompact` event onto its internal stream (`src/core/query.ts:421`), but the turns route's `mapServerStreamEvent` currently returns null for it (deferred to M4+ per the file header comment). Integration tests that need to observe microcompaction must inspect the provider's call-2 messages array directly (cleared `[Tool result cleared`-prefixed placeholders) rather than relying on an SSE event. M9 visual polish is the natural place to wire it through.
