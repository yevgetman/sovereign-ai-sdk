## Critical fix landed post-T7 (2026-05-15)

**Bug:** Latent M3-era correctness bug in the Go TUI. The TUI subscribed to the SSE stream exactly ONCE in `New()` (`packages/tui/internal/app/app.go:101-103`) and never reconnected. The server closes the SSE response body and disposes the per-session bus on every `turn_complete` / `turn_error` (`src/server/routes/events.ts:63-74` ‚Äî by design, the bus is per-turn). All turns AFTER the first delivered 202 from POST `/sessions/:id/turns` (the bus was recreated on the publish side) but the events were silently dropped client-side because the original SSE subscription had already terminated and was never replaced. The user saw their input echo, the dim "‚Ä¶thinking" placeholder, and then nothing ‚Äî a `[stream closed]` line landed once at end-of-turn-1 and that was it.

**Discovery path:** Autonomous probe against the production server (`mock` provider, two consecutive POST `/sessions/:id/turns` calls on a single GET `/sessions/:id/events` subscription). The probe fired before the user's three M6 manual smoke scenarios had a chance to surface the bug. Caught it pre-user-smoke.

**Fix (`3365fb3`):** In `app.go`'s `sseDoneMsg` handler, re-Consume a fresh SSE stream against the current `m.sessionID` (which may have pivoted via `/compact` or `compaction_complete`). Skip when the app context is cancelled (user pressed ESC / Ctrl+C) or `baseURL == ""` (render-only test fixtures). Drop the `[stream closed]` transcript line ‚Äî meaningful when the design was single-turn-per-launch, but with reconnect it's noise after every turn. Bubble Tea correctness: `waitEvent` is a value-receiver method (not a closure), so reassigning `m.events` / `m.errs` in Update and returning `m.waitEvent` lets the new channels take effect on the next wait ‚Äî verified by reading `app.go:132-151` before applying the fix.

**Test (`TestApp_reconsumesSSEAfterTurnComplete`):** Pins the multi-turn contract by driving two consecutive turns through `teatest.NewTestModel` against an `httptest.Server`. Server's SSE handler counts connections and serves two distinct SSE responses for the two turns; subsequent reconnects (after turn 2's `turn_complete`) hold open via `<-r.Context().Done()` so `srv.Close()` doesn't block on test teardown. Final assertion bounds connection count to `[2, 3]` ‚Äî 2 = bare minimum (1 per turn), 3 = steady state (initial + 1 reconnect after each turn_complete), > 3 would indicate a tight reconnect loop bug. RED-GREEN verified: stashed the fix, the new test timed out at 3s waiting for `TURN_TWO_REPLY` showing `[stream closed]`; restored the fix, PASS in 0.32s.

**Existing test fixture adjustment:** `TestApp_consumesMultipleEventsFromSingleConnection` ‚Äî pre-existing handler emitted 3 events including `turn_complete` then returned. With reconnect, that handler would be invoked in a tight loop. Tightened the handler to hold connection #1 open via `<-r.Context().Done()` after emitting (so all 3 events are PROVEN to arrive on connection #1) and changed the assertion from `connectionCount == 1` to `connectionCount <= 2`. The invariant being guarded ‚Äî "all 3 events arrive on a single connection, not via per-event reconnect" ‚Äî is preserved.

**Compaction interaction (no regressions):** Mid-turn compaction events arrive on the SAME bus as the rest of the turn's events (the bus is captured into `bus` at `runTurnInBackground`'s start). The fix does NOT reconnect on `compaction_complete` ‚Äî `m.sessionID` is mutated and the next `sseDoneMsg` (after the post-pivot `turn_complete`) triggers the re-Consume against the new id. `TestApp_compactionCompleteSSEPivotsSession` (M6) and `TestApp_compactSlashRoutesToCompactEndpoint` (M6) both still pass post-fix.

**Impact on M6 smoke viability:** The bug blocked all multi-turn `--ui tui` usage including Scenarios 2 and 3 of the M6 manual smoke (proactive compaction needs sustained turn count; overflow recovery needs a long tool-using turn that triggers re-issue against the new session). Scenario 1 (explicit `/compact`) was technically completable to the compact-marker step but the post-compact normal-turn dispatch was silently broken. Post-fix all three scenarios are actionable end-to-end.

## Behavioral notes worth knowing next session

1. **Compaction creates a new session id; the TUI hops `m.sessionID`.** Per ADR M6-01: every compaction (proactive, overflow recovery, or explicit `/compact`) creates a fresh child session id. The TUI updates `m.sessionID` on receiving either the `compaction_complete` SSE event OR the JSON response from `POST /sessions/:id/compact`. Subsequent POSTs (turns, approvals, further compactions) route to the new child. SessionDb persists parent‚Üíchild lineage via `recordCompactionLineage` for audit; the SSE subscription stays on the parent id (the bus is keyed on parent and continues to surface post-compaction events under the parent id).

2. **Single retry on context-overflow.** Per ADR M6-02: first overflow ‚Üí compact + retry once. Second overflow ‚Üí `turn_error`. The proactive + recovery branches operate on independent budgets ‚Äî a proactive compaction earlier in the same turn does NOT prevent the recovery branch from firing, so a single turn can emit TWO `compaction_complete` events. The TUI's compaction handler is idempotent ‚Äî it pivots `m.sessionID` to the LATEST `activeSessionId` (the recovery child id wins over the proactive child id when both fire).

3. **`POST /sessions/:id/compact` is synchronous.** Per ADR M6-03: the explicit verb runs compaction inline and returns the JSON `CompactResult` body. No SSE event fires for this path ‚Äî the JSON response IS the notification. Errors return JSON-shaped responses (400/404/500) with `{ error: <message> }` envelope matching sibling routes.

4. **Microcompaction triggers based on tool-result token share.** `shouldMicrocompact(history, config)` returns true when the per-part `tool_result` blocks exceed `config.triggerThresholdPct` of the estimated total context. Default 40%. The microcompact pass replaces older tool_result content with `[Tool result cleared`-prefixed placeholders, keeping the most recent `config.keepRecent` (default 5) intact. Fires inside the turn loop AFTER `runTools` and BEFORE the next provider call ‚Äî mutates the in-flight history array but never persists. Default `compactableTools` covers Bash + WebFetch + Read; configurable via `userSettings.microcompaction` (though `compactableTools` is not in the schema today ‚Äî M6 left it as a code default).

5. **Proactive threshold stored as percentage.** `userSettings.compaction.proactiveThresholdPct` is a 1..99 integer (human-friendly editing); `buildRuntime` divides by 100 to get the fraction `shouldCompactProactively` expects. Default 0.75 from the function's own default. The compactor self-guards against the frozen system prompt alone exceeding the threshold (`compactor.ts:177-183`) ‚Äî without that guard, compaction would loop indefinitely because the child still has the same system prompt.

6. **Same-provider summarization (no auxiliary model).** Per inline decision M6-06: the `summarize` callback uses `runtime.resolvedProvider.transport.stream(...)` with `runtime.model`. No auxiliary cheap-model routing. Auxiliary selection is Phase 15 / M8 polish.

7. **The wrapped-runtime test pattern.** The M6 integration suite (`tests/cli/tuiLauncherIntegration.test.ts:M6 describe block`) uses a `buildWrappedRuntimeModule` factory to inject `pre`/`post` hooks before/after `buildRuntime` runs. The wrapper captures the function reference BEFORE the mock is mounted so the wrapper's recursion into "the real" `buildRuntime` resolves to the production implementation rather than re-entering itself (avoids stack overflow). This pattern is the right shape for any future integration test that needs to inject behavior the launcher's CLI-shaped opts bag doesn't expose.

8. **The microcompact event is NOT yet on the wire.** `query()` emits a `microcompact` event onto its internal stream (`src/core/query.ts:421`), but the turns route's `mapServerStreamEvent` currently returns null for it (deferred to M4+ per the file header comment). Integration tests that need to observe microcompaction must inspect the provider's call-2 messages array directly (cleared `[Tool result cleared`-prefixed placeholders) rather than relying on an SSE event. M9 visual polish is the natural place to wire it through.

9. **`CompactResult.noOp: true` indicates "nothing to compact" (item #36).** When `compactSession`'s `head` is empty (small history fits in tail budget), the function returns early without writes ‚Äî `parentSessionId === newSessionId`, `noOp: true`, no `recordCompactionLineage` call. The flag is plumbed through `runtime.compact` ‚Üí server routes ‚Üí Go TUI's `compactCompleteMsg` and SSE handler. TUI renders `‚îÄ nothing to compact (history already fits)` (no session-id pivot). Recovery branch surfaces `turn_error` if compaction returns no-op (a no-op compact wouldn't help with overflow). Tests that construct minimal sessions need to seed enough history to clear the `selectTailStart` floor (‚â•5 messages with ‚â•4K tokens of compactable content above the tail budget).

10. **Compaction summary is `role: 'assistant'` BUT a synthetic `user` bridge gets prepended when the tail starts with assistant (item #34).** `compactSession` checks `tail[0]?.role` after building the new history; on `'assistant'`, prepends `{role: 'user', content: [{type: 'text', text: '(continuing from summary)'}]}` between summary and tail to maintain Anthropic's strict alternation. Mock provider doesn't care; real Anthropic 400'd on the original shape. The bridge is included in the returned `result.tail` so both consumers (server route via `hydrate()` and terminalRepl via `result.tail` directly) receive the alternation-safe history without caller-side changes.

11. **Real Anthropic `BadRequestError` on overflow is `'prompt is too long: N tokens > 200000 maximum'` (item #35 verified).** The matcher's `'prompt is too long'` substring catches it. `tests/providers/errors.test.ts` (12 cases) pins the verified format alongside synthetic + 413 + negative shapes. If a future Anthropic SDK change reformats the error message and the substring stops matching, T4's overflow recovery will silently fail (turn_error instead of recovery); the test would catch that on the next CI run.

12. **Inherited notes from prior snapshots** (still valid):
