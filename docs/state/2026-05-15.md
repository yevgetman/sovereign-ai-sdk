## Scope decisions noted (M7-04, M7-07 ‚Äî not ADRs)

Two scope decisions surfaced during M7 that are noted here for clarity but were not promoted to ADRs because they are about scope/sequencing rather than runtime architecture:

## Behavioral notes worth knowing next session

1. **Per-session SessionContext is the M7 pattern.** Anything that needs to live for the lifetime of a session and tear down at session end goes onto `SessionContext` and constructs in `buildSessionContext`. The lazy-build-and-cache `getSessionContext` keeps construction local to first-reference. Disposal walks the map first inside `runtime.dispose()` per ADR M7-08. M8/M9 additions of similar shape (e.g., per-session router audit log if M8 takes it that way) should follow the same pattern.

2. **Trace writer is rebuilt on compaction (ADR M7-02).** When the turns route pivots `sessionId` to a compaction child id, the next `getSessionContext(childId)` call constructs a fresh `SessionContext` with a fresh trace writer pointed at `<harnessHome>/traces/<childId>.jsonl`. The parent's writer stays open until the parent is disposed. `sov trace show <childId>` reads from the child's file; `sov trace show <parentId>` reads from the parent's. Lineage walking across the two is a future audit-tool concern, not a runtime concern.

3. **Trajectory is disposal-driven, not per-turn (ADR M7-03).** A session that hasn't been disposed has no trajectory record ‚Äî the writer fires from `disposeSessionContext`. The `Runtime.dispose()` shutdown walk picks up every live entry. Process crashes lose trajectories for sessions that haven't been disposed; this is the cost of disposal-driven writes, and is mitigated by fire-and-forget semantics so disposal itself completes even if the write fails.

4. **Learning + review observation is direct-call, not bus-subscribed (M7-04 + ADR M7-06).** Orchestrator + scheduler + turns route fire `ctx.<subsystem>?.<method>(...)` optional-chain calls at the relevant moments. The `DaemonEventBus` is plumbing for cross-process consumers (future daemon-mode); no in-process subscriber exists in M7. If a future milestone needs cross-process observation, it can attach to the bus without re-plumbing the in-process path.

5. **`session_summary` SSE event lives on the disposal bus, not the per-turn bus.** Per-turn buses are short-lived (close at `turn_complete` / `turn_error`). The disposal bus is whatever the caller of `runtime.disposeSession(sessionId, { bus })` supplied ‚Äî typically the long-lived top-level SSE subscription. Inside `runtime.dispose()` at process shutdown there's no bus available (the SSE consumer has disconnected); the disposal walk logs the summary to stderr ONLY if `totalDispatched > 0` so the zero-case doesn't spam tests.

6. **`onUserTurn` is fired AFTER user-message persist, BEFORE hydrate-and-stream.** Same semantic moment as terminalRepl (`src/ui/terminalRepl.ts:1283,1290`). The T6 follow-up wiring was load-bearing for `userTurnsForMemoryReview` and `synthesizerEveryN` ‚Äî these settings are silently inert if the call is missing. The integration smoke pins this via the onUserTurn spy.

7. **Backlog #38 carry-forward.** `reviewAutoPromoteMemory` / `reviewAutoPromoteSkills` are read by `MemoryProposeTool` and `SkillProposeTool` off `ctx.review*` but `buildSessionContext` doesn't thread them onto the `parentToolContext` snapshot, and `buildSessionToolContext` doesn't thread them onto the per-turn `ToolContext` either.
