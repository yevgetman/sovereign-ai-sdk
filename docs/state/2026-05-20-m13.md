## Behavioral notes worth knowing next session

1. **Bare `sov` boots Go TUI; missing binary = exit 1.** Per ADR M13-01: when `findTuiBinary()` returns null, `src/main.ts` writes the install command to stderr and exits 1. No fallback surface. The install command is the canonical `bun pm -g trust @yevgetman/sov && sov upgrade`.
2. **`--ui` flag and `SOV_UI` env no longer exist.** Per ADR M13-03: Commander rejects `--ui <anything>` as an unknown option (smoke scenario 03 verifies). `SOV_UI=repl` in the environment is silently ignored — no parsing, no warning. Author needs to scrub shell rc if it had `export SOV_UI=tui` (harmless either way).
3. **`SOV_NO_DEPRECATION_WARNING` no longer exists.** The M12 deprecation warning is gone, so its escape-hatch env var is gone with it (also per ADR M13-03).
4. **`ui.surface` config field is rejected by schema.** Per ADR M13-02: Zod's `.strict()` mode on `UiSchema` means an existing config with `ui.surface: "repl"` will fail to parse at boot. No migration preprocessor — the author edits their own config.
5. **Readline-asker permission path is gone.** `canUseTool` in `src/permissions/canUseTool.ts` now uses `serializeAskUser` + `previewToolInput` from the slim post-M13 `src/permissions/prompt.ts`. The `buildReadlineAsker` + `parseAskResponse` exports — and the `import { type ModalRow, withModal } from '../ui/modal.js'` they depended on — are gone. The `withModal` modal helper itself survives (used by autocomplete and other live consumers).
6. **No deprecation warnings, no fallback, no surface resolution.** The post-M13 `src/main.ts` does one thing: launch the TUI binary. There is no `resolveSurface(...)` call, no `effectiveSurface` variable, no fallback branch. Anything that used to live in those code paths is either deleted or moved into `tuiLauncher.ts`.
7. **`sov chat` deprecation message is unchanged.** The `sov chat` deprecated subcommand's stderr notice at `src/main.ts` is a separate thread (introduced pre-M11) and untouched by M13. A future cleanup pass can retire it.
