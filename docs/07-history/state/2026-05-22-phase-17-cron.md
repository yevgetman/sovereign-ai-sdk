## Where we are

Phase 17 introduces **scheduled agent runs**: short-lived fresh-session conversations triggered by a tick loop embedded in the runtime's lifecycle, delivered to any channel, with optional skill chaining and pre-agent scripts. The pattern is **Hermes-style fire-and-forget**, deliberately contrasted with Phase 13.5's Fry-style stateful missions — each cron run starts from a clean session, runs to completion, writes its output to an outbox, and disposes. No persistent mission state, no retries, no resumption.

The architecture is purely additive: no existing surface removed, no foreground refactor. A new `src/cron/` directory carries the subsystem (types, schedule parser, jobs CRUD, runner, executor, production glue). `buildRuntime` grows a single optional field (`cronRunner`) and one new option (`cronEnabled`). `sov cron <subcommand>` lands as a sibling to `learning`, `chat`, `drive`, `dispatch`. The 60-second tick loop runs only while `sov` is alive — v0 deployment is "keep `sov` open in a long-lived terminal pane."

The user kicked off Phase 17 at the start of this session asking to build out cron/scheduled jobs against the harness plan. Subagent-driven development per T1 → T9; this is T10 (docs close-out); T11 (cut v0.3.0) follows.

## What shipped

### New `src/cron/` subsystem (~640 LoC across six files)

- **`src/cron/types.ts`** — `ScheduleKind` (relative / interval / cron / ISO), `Job` record, `JobsFile` envelope.
- **`src/cron/schedule.ts`** — `parseSchedule` (regex for `30m` / `every 2h`, ISO via `Date.parse`, cron via `cron-parser@^4.9.0`), `computeNextRun` (one-shot vs recurring semantics; relative + iso are one-shot returning `null` after first run, interval + cron are recurring).
- **`src/cron/jobs.ts`** — `addJob`, `listJobs`, `getJob`, `pauseJob`, `resumeJob`, `deleteJob`, `recordJobRun`. Atomic temp+rename persistence to `<harnessHome>/cron/jobs.json` (PID + timestamp in the temp filename to avoid collisions across concurrent writes).
- **`src/cron/runner.ts`** — `CronRunner` class. Constructor takes `harnessHome`, `now()` (for fake-clock tests), `runJob(job)` (the executor), optional `tickIntervalMs`. `start()` spawns `setInterval(..., 60_000).unref?.()` so the timer doesn't keep test processes alive. `tick()` guards via the `<harnessHome>/cron/.tick.lock` directory (atomic `mkdirSync` — fails if another process holds it). `runDueJobs()` filters `jobs.filter(j => j.enabled && j.nextRunAt !== null && j.nextRunAt <= now())`, runs them sequentially, persists `lastRunAt` + `lastResult` + the recomputed `nextRunAt` via `recordJobRun`.
- **`src/cron/execute.ts`** — `buildCronJobExecutor` factory. Wires together skill expansion + pre-agent script + the agent loop + delivery. Returns a `(job) => Promise<CronRunResult>` closure that's plugged into `CronRunner.runJob`. Prompt assembly order: `## Script output\n\n<stdout>` → `\n\n---\n\n` → expanded skills (joined with the same separator) → `\n\n---\n\n` → operator prompt. Script timeouts default to 120s; stdout capped at 16 KiB.
- **`src/cron/wiring.ts`** — production glue from `buildRuntime` to the executor. `createProductionCronRunner(runtime, harnessHome)` constructs the executor with three real deps: `runAgent` (mints a fresh `metadata.kind='cron'` session, builds `AgentRunner` against `runtime.toolPool` minus `SUBAGENT_EXCLUDED_TOOLS`, drains the generator, extracts final assistant text via `content.filter(type === 'text')`), `expandSkills` (looks each name up in `runtime.skills.byName`, joins via `expandSkillPrompt` + the separator), `runScript` (`spawnSync` with timeout + interpreter inference: `.py → python3`, `.ts/.js → bun`, `.sh → bash`, otherwise direct exec). `canUseTool` is `mode: 'default'` with an auto-deny `ask` callback — matches `sov drive` headless policy.

### CLI surface (`src/cli/cronCommand.ts` + Commander subcommands in `src/main.ts`)

- **`sov cron add`** — `--schedule <spec> --prompt <text> [--deliver <target>] [--skills s1 s2 ...] [--script <path>] [--script-timeout-ms <n>]`. Delegates to `runCronAdd` which calls `parseSchedule` + `addJob`.
- **`sov cron list`** — formatted table of all jobs (id, status, next run, prompt preview).
- **`sov cron show <id>`** — full job JSON.
- **`sov cron pause <id>`** / **`sov cron resume <id>`** — flip the `enabled` flag.
- **`sov cron delete <id>`** — remove the job from `jobs.json`.
- **`sov cron tick`** — manually fires `runDueJobs()` once (debug). Boots `buildRuntime({ cronEnabled: false })` so the manual tick doesn't compete with the embedded loop, builds `createProductionCronRunner` directly, calls `runner.runDueJobs()`, disposes.
- **`sov cron run <id>`** — currently fires `runDueJobs()` (same as `tick`); the `<id>` arg validates the job exists. Strict single-job-fire is in the follow-ups list below.

### Delivery hook (`src/channels/delivery.ts` + `src/channels/types.ts`)

(Previously `cron_create` / `cron_list` / `cron_delete`.) Sub-agents spawned from a cron run cannot themselves call any of these six tool names — prevents cron jobs from manipulating cron state.

## Behavioral notes worth knowing next session

1. **The tick loop fires only while a `sov` runtime is alive.** No standalone daemon. v0 deployment pattern: keep `sov` (TUI / `sov drive` / `sov dispatch`) running in a long-lived terminal pane. If `sov` exits, no jobs run until it's restarted. Documented in the plan's R2 risk; surfaced via the CLI (`sov cron list` shows a status line warning the user of this constraint — see T8 implementation if you adjust the format).
2. **Permission policy is auto-deny on `ask` fall-through.** Matches `sov drive` headless mode. Cron jobs run with `mode: 'default'`, so the layered allow/deny rules fire normally; only when a tool's self-check returns `ask` does the auto-deny kick in. No bypass flag in v0.
3. **Skills inject into the user message, not the system prompt.** Matches existing harness convention (M8+). The spec's "system prompt" phrasing predates M8 — Q3 in the plan's decision table locks this. `expandSkillPrompt` is the same code path used by interactive `/skill-name` invocations.
4. **Script paths resolve under `<harnessHome>/cron/scripts/`** for relative paths; absolute paths pass through. Interpreter inferred from extension (`.py → python3`, `.ts/.js → bun`, `.sh → bash`, otherwise direct exec). Default timeout 120s; output capped at 16 KiB to prevent prompt-side blow-up. No sandbox in v0 (trust boundary matches `sov init` — scripts run with full env credentials).
5. **Recursion guard: cron-spawned sub-agents can't call cron CRUD tools** (`cron_add` / `cron_list` / `cron_show` / `cron_pause` / `cron_resume` / `cron_delete`). The exclusion set is in `src/agents/exclusions.ts` (`SUBAGENT_EXCLUDED_TOOLS`); the wiring filters them out before constructing the cron-job `AgentRunner`.
6. **`[SILENT]` prefix is case-insensitive, first-8-chars-after-trim.** The check is literally `content.trimStart().toLowerCase().startsWith('[silent]')`. Only inspects the first line by virtue of `startsWith` semantics. Documented contract — narrow on purpose; the model needs an unambiguous signal it can emit when it has nothing worth delivering.
7. **`sov cron run <id>` currently fires any due jobs** rather than only the named job. The id arg validates that the job exists. Strict single-job-fire is in the follow-ups; today the operator can `pause` the other jobs and then `run` to get single-job behavior.
8. **Sessions are tagged `metadata.kind = 'cron'`** with `cronJobId` on the metadata. No cleanup sweep yet — session DB will accumulate cron sessions over time; cleanup is deferred (see follow-ups).
9. **The default 60-second tick uses `.unref?.()`** so test processes don't hang on the interval. `cronEnabled: false` in `buildRuntime` opts out entirely — used by the manual `sov cron tick` / `sov cron run` paths so they don't compete with the embedded loop, and by tests that don't want to deal with timer cleanup.

## Open follow-ups

1. **Strict single-job-fire for `sov cron run <id>`** — currently fires `runDueJobs()`; should fire just the named job regardless of `nextRunAt`. Includes partial-id matching for ergonomics.
2. **Long-running cron daemon** — separate `sov cron daemon` process that runs the tick loop without a parent `sov` session. Defer until v0 deployment pattern proves insufficient.
3. **Per-job model override** — `--model <provider:model>` on `sov cron add`. Today every job uses the runtime's default model.
4. **Per-job `allowedTools` scoping** — cron-wide exclusion set only in v0. Per-job allowlist or denylist is the next refinement.
5. **Retry-on-failure with backoff** — failed jobs simply record `ok: false` in `lastResult`; the schedule's next fire is unaffected. Exponential backoff + max-retry would help transient model errors.
6. **Webhook-triggered jobs** — fire on external HTTP POST instead of a schedule. Requires the runtime's `startServer` to expose a `/cron/trigger/<id>` route + auth model.
7. **Cleanup sweep for old cron sessions** — purge `metadata.kind='cron'` sessions older than N days. Today session DB will accumulate them indefinitely.
8. **TUI surface for cron management** — CLI-only in v0. A TUI picker card (per the M11.5 pattern) would let operators add/pause/delete jobs interactively.
9. **`[SILENT]` semantics could extend** — currently first-line-only by virtue of `startsWith`. Could become "any line of the output is `[SILENT]`" or "`[SILENT-DELIVERY]` vs `[SILENT-ALL]`". Defer until the contract bites someone.

## How it works now

```bash
# Schedule a job to summarize today's AI news at 9am daily
sov cron add \
  --schedule "0 9 * * *" \
  --prompt "Summarize today's AI news" \
  --deliver local

# Schedule a one-shot in 30 minutes
sov cron add --schedule "30m" --prompt "Check on the build" --deliver local

# Schedule every 2 hours with a pre-agent script + chained skills
sov cron add \
  --schedule "every 2h" \
  --prompt "Look at the deploy status" \
  --script "check-deploy.sh" \
  --skills observability incident-response \
  --deliver local

# Schedule for a specific ISO timestamp (one-shot)
sov cron add --schedule "2026-05-23T17:00:00Z" --prompt "EOD wrap-up" --deliver local

# List all jobs
sov cron list

# Show one job's full record
sov cron show <id>

# Pause / resume / delete
sov cron pause <id>
sov cron resume <id>
sov cron delete <id>

# Manual fire of any due jobs (debug)
sov cron tick

# Same, with id validation that the job exists (v0: still fires all due)
sov cron run <id>
```

Outbox lands at `<harnessHome>/cron/outbox/<jobId>/<timestamp>.txt`. Each run writes one file with the final assistant text. `[SILENT]` prefix in the assistant output suppresses the file write while still recording `lastResult.ok = true`.
