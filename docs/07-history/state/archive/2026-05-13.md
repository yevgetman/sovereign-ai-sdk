### Post-manual-smoke fixes

Two Critical bugs were not caught by either spec-compliance or code-quality reviews and only surfaced during the user's visual smoke. Both shipped today:

1. **Tool-using turns truncated after the preamble** (commit `e1a9cba`). `turns.ts` mapped every internal `message_stop` to `turn_complete`, including the intermediate ones in a multi-call tool-use turn. The events route closed the SSE stream on the first `turn_complete` so the tool never ran (or its output never reached the client). Fixed by switching to manual `await stream.next()` iteration to capture the generator's `Terminal` return value as the real end-of-turn signal, plus mapping assistant + tool-result Messages into `tool_use_start` / `tool_use_done` / `tool_result` SSE events.

2. **`buildRuntime` loaded zero permission settings** (commit `6c782a5`). Server-side `query()` fell through to runtime-default `'ask'` mode and emitted `permission_request` events that no client could approve, so any tool-using turn hung indefinitely on `…thinking`. The same prompt in `--ui repl` worked because `terminalRepl.ts` honors `~/.harness/config.json`'s `permissionMode`. Fixed by mirroring terminalRepl's permission cascade in `buildRuntime` (explicit option → permission_settings file → user config.json → `'default'`), building `canUseTool` with the right mode + rule layers, wrapping with `redactSecretsTransformer`, and passing the resulting `canUseTool` into the `query()` call. `ask()` is a placeholder that denies with guidance text; M5+ will wire the proper SSE/POST round-trip.

3. **TUI silent failure modes** (commit `8ed09fc`). Added `turn_error` handler, `…thinking` placeholder cleared by first response event, visible `turn_complete` marker. Also added `permission_request` handler for defense in depth (commit `6c782a5`).

## What does NOT work / known gaps for M4+

These are deliberate M3 scope-outs documented in the spec:

1. **No interactive permission prompt in the TUI.** The server's `ask()` callback is a 'deny' placeholder. If a user changes `permissionMode` from `bypass` to `default`, the Go TUI will show a yellow warning (via the new `permission_request` handler) instead of approving — they'd need ESC and a config change. M5 will wire the full `permission_request` → POST `/approvals/:requestId` round-trip with an interactive modal in the TUI.

2. **No 24-prereq subsystems wired.** M3 was the "bare turn" milestone. None of the 24 subsystems from `docs/08-roadmap/backlog/phase-16-rebuild-prereqs.md` are wired through the new foreground yet. Specifically missing (will land M4–M8):

   - Preflight checks
   - Full CLI flag forwarding
   - Hooks (PreToolUse / PostToolUse / Stop)
   - Permission modal (see #1)
   - Sub-agent scheduler
   - Compactor + microcompaction + context-overflow recovery

   - Trace writer + trajectory capture + learning observer + review manager
   - Local-model router, capture/replay, `@file:path` expansion, subdirectory hints, skill-as-slash-command, skill visibility filtering, goodbye summary, stall detection, tool-result expand registry

3. **Visual polish deferred to M9.** Tool cards are minimal placeholders. No markdown rendering for assistant text yet, no syntax highlighting on code or diffs, no inline-diff hunk navigation, no slash autocomplete popup, no mouse, no light/dark theme switch, no custom theme loading.

4. **3 Go tests skipped with `t.Skip`.** `TestApp_rendersTurnErrorVisibly`, `TestApp_showsThinkingIndicatorOnEnter`, `TestApp_thinkingClearedByFirstResponseEvent` — the teatest harness's `WaitFor` polling doesn't catch the rendered text within its 3s window despite the handler being reached. ## Retrospective — what went wrong with the M3 review process

Two Critical bugs (e1a9cba, 6c782a5) slipped through both spec-compliance and code-quality reviewers and only surfaced during the user's visual smoke. Both involved silent failure modes in inter-process flows:
- `turns.ts` emitting redundant `turn_complete` events the SSE consumer closed on
- `buildRuntime` not loading permission settings, causing silent server-side hangs

The unit tests covered the in-process Hono request shape, not the multi-step query() loop or the runtime-config resolution. The code-quality review's "approved" framing was based on file structure + test counts, not on user-flow simulation.

**Lesson for M4+:** The code-quality review process should include a "real-user smoke" step using a representative tool-using prompt with a non-bypass permission config, against the actual TUI (not just the headless wire). Filed as a process-level follow-up.
